# 택배 계약 플랫폼 도메인

## 개요

이 프로젝트의 핵심 도메인은 화주와 택배 대리점을 직접 연결하는 택배 계약 플랫폼이다.
서비스명은 기획 자료 기준 `택배왕`이다.

택배왕은 전화, 소개, 직접 영업에 의존하던 택배 계약 과정을 디지털화한다. 화주는 여러 대리점이 제안한 단가와 서비스 조건을 비교해 계약하고, 대리점은 신규 화주 물량을 확보할 수 있다.

## 택배 운영 흐름

일반적인 택배 흐름은 다음과 같다.

```text
화주 -> 기사 집하 -> 집하 터미널 -> 허브 터미널 -> 도착지 터미널 -> 기사 배송 -> 소비자
```

계약은 보통 화주와 대리점 사이에 체결된다.

- 화주: 물건을 보내는 판매자, 쇼핑몰, 기업 등
- 대리점: 택배사의 지역 영업 거점
- 택배사: CJ, 한진, 롯데 등 물류 네트워크를 운영하는 회사
- 배송기사: 대리점 소속 또는 연계 기사로 실제 집하와 배송을 수행

화주는 자신의 물량을 처리할 대리점과 계약한다. 대리점은 월 예상 물량, 단가, 서비스 조건을 제안하고, 계약이 성사되면 해당 물량의 집하와 배송을 수행한다.

### 지역 대리점 구조 예시

같은 택배사라도 동네마다 담당 대리점이 다를 수 있다. 예를 들어 안산에는 일동, 사동, 본오동, 이동, 고잔동, 선부동 등 여러 생활권이 있고, 각 지역별로 CJ, 한진, 롯데, 로젠 대리점이 따로 존재할 수 있다.

예시 대리점:

- CJ 일동대리점
- CJ 본오대리점
- 한진 사동대리점
- 롯데 고잔동대리점
- 로젠 선부동대리점

따라서 화주는 단순히 "CJ와 계약한다"가 아니라, 실제로는 자신의 지역과 물량을 처리할 수 있는 특정 대리점을 찾아 계약해야 한다.

## 기존 시장 문제

택배 계약 시장은 규모가 크지만 계약 방식은 여전히 아날로그에 가깝다. 전화 문의, 지인 소개, 네이버 카페 홍보, 직접 영업 같은 방식에 의존하는 경우가 많다.

### 화주 문제

- 단가와 서비스 조건을 객관적으로 비교할 기준이 부족하다.
- 어느 택배사와 대리점이 적합한지 찾기 어렵다.
- 상담, 서류, 방문 중심 계약으로 계약 과정이 느리다.
- 물량이 적거나 신규 화주인 경우 협상력이 낮다.
- 근처 대리점에 연락해 보는 수준으로 시장을 탐색하다 보니 더 좋은 조건을 놓칠 수 있다.

예를 들어 안산 일동에서 옷가게를 운영하는 사장이 월 800박스를 보낸다고 가정한다. 이 사장은 `60사이즈 박스 단가`, `토요일 배송 가능 여부`, `냉장 가능 여부`, `반품 처리 조건` 등을 확인하기 위해 CJ 일동대리점에 먼저 전화하고, 가격이 마음에 들지 않으면 CJ 본오대리점, 한진 사동대리점, 롯데 고잔동대리점, 로젠 선부동대리점에 다시 전화해야 한다.

이 과정의 핵심 문제는 비교 비용이다. 화주는 여러 대리점의 조건을 한 번에 볼 수 없고, 대리점마다 전화로 같은 정보를 반복해서 설명해야 한다. 결국 가격뿐 아니라 서비스 조건까지 직접 표로 정리해야 비교가 가능하다.

### 대리점 문제

- 신규 화주를 직접 만날 기회가 적다.
- 먼저 연락이 오거나 소개받지 않는 이상 새로운 화주와 연결되기 어렵다.
- 대형 대리점이나 기존 거래 관계가 강한 대리점에 비해 노출 기회가 적다.
- 지역을 돌며 영업하거나 온라인 커뮤니티에 글을 올리는 방식은 시간 비용이 크다.
- 화주의 예상 물량, 박스 크기, 상품 특성 등을 사전에 알기 어려워 대응이 비효율적이다.

### 시장 구조 문제

- 계약 시스템은 매년 커지는 택배 물량을 따라가지 못하고 있다.
- 가격 형성이 불투명하고 단가 왜곡이 발생할 수 있다.
- 택배사, 대리점, 화주 사이의 정보 비대칭이 크다.
- 기존 방식은 시장 전체의 합리적 경쟁을 만들기 어렵다.

## 택배왕 솔루션

택배왕은 화주와 대리점을 연결하는 입찰형 택배 계약 플랫폼이다.

### 기본 흐름

1. 화주가 플랫폼에 계약 신청 정보를 등록한다.
2. 등록 정보에는 발송 지역, 예상 발송 물량, 박스 크기, 물건 정보, 픽업 희망 시간 등이 포함된다.
3. 해당 정보가 근처 또는 담당 가능 지역의 대리점에 전달된다.
4. 대리점은 택배 1건당 가격과 서비스 조건을 제안한다.
5. 여러 대리점이 비공개 입찰 방식으로 참여한다.
6. 화주는 가격, 서비스 조건, 평판, 리뷰 등을 비교한다.
7. 화주는 적합한 대리점을 선택하고 전자계약을 진행한다.
8. 계약 후 대리점 소속 기사들이 집하와 배송을 수행한다.

### 안산 옷가게 계약 예시

화주가 다음 조건으로 계약 요청을 등록한다.

```text
지역: 경기도 안산시 일동
월 예상 물량: 800박스
물품 종류: 일반 의류
박스 크기: S박스 또는 60사이즈 중심
픽업 희망 시간: 오전 9시 ~ 오후 6시
필요 조건: 토요일 배송, 반품 처리 조건 확인
```

택배왕은 이 요청을 처리 가능성이 있는 주변 대리점에 전달한다.

- CJ 일동대리점
- CJ 본오대리점
- 한진 사동대리점
- 롯데 고잔동대리점
- 로젠 선부동대리점

각 대리점은 다른 대리점의 제안 단가를 보지 못한 상태에서 자신의 조건을 제출한다. 화주에게는 여러 제안이 비교 가능한 형태로 정리된다.

```text
CJ 일동대리점   평점 4.8 -> S박스 2,050원 / 토요일 가능 / 냉장 가능
한진 사동대리점 평점 4.6 -> S박스 1,950원 / 토요일 불가
롯데 고잔동대리점 평점 4.7 -> S박스 2,100원 / 반품 무료
```

화주는 최저가만 보는 것이 아니라 픽업 가능 시간, 토요일 배송, 반품 조건, 대리점 평점, 리뷰까지 함께 비교해 선택한다.

### 주요 도메인 객체 후보

- 계약 요청: 화주가 등록하는 배송 계약 수요
- 제안: 대리점이 계약 요청에 대해 제출하는 단가와 조건
- 대리점 후보: 지역, 택배사, 처리 가능 물량, 서비스 조건을 기준으로 매칭되는 대리점
- 비교 항목: 단가, 평점, 리뷰, 토요일 배송 여부, 냉장 가능 여부, 반품 조건, 픽업 시간
- 계약: 화주가 특정 제안을 선택해 성사된 거래

### 기본 매칭 기준

- 지역: 화주의 사업장 주소와 대리점 담당 지역 또는 픽업 가능 지역
- 물량: 월 예상 물량을 대리점이 감당할 수 있는지 여부
- 물품: 일반 의류, 식품, 냉장/냉동, 파손 위험 상품 등
- 박스 크기: S박스, 60사이즈 등 대리점 단가 기준에 맞는 규격
- 서비스 조건: 토요일 배송, 반품 회수, 픽업 시간, 냉장 가능 여부 등

### 추천 도메인

추천은 `Recommendation` 도메인으로 관리한다. 초기 구현은 화주에게 대리점을 추천하지만, 이후 대리점에게 화주 추천, 대리점에게 배송기사 추천으로 확장될 수 있어야 한다.

추천 구조는 다음 기준을 가진다.

- 추천 요청자: 추천을 조회하는 주체다. 예: 화주, 대리점
- 추천 대상: 추천 결과로 내려가는 대상이다. 예: 대리점, 화주, 배송기사
- 추천 목적: 어떤 업무 흐름에서 추천하는지 표현한다. 예: 계약 요청 대리점 매칭, 대리점 영업 화주 발굴, 배송기사 배정
- 추천 사유: 사용자가 추천 결과를 납득할 수 있도록 이전 계약, 같은 권역, 담당 가능 지역 같은 이유를 함께 제공한다.

초기 화주 대상 대리점 추천 기준은 다음과 같다.

- 이전 계약 이력이 있는 대리점을 우선 추천한다.
- 화주의 주요 지역을 담당 가능 지역으로 가진 대리점을 추천한다.
- 화주의 주요 지역과 대리점 주 담당 지역이 같은 경우 점수를 추가한다.
- 추천 결과는 DB에 저장하지 않고 조회 시점에 계산한다. 추천 로그, 클릭, 선택 이력 저장은 추천 품질 개선 단계에서 별도로 확장한다.

### 입찰과 비교 규칙

- 대리점 제안은 화주에게만 공개되는 비공개 제안으로 취급한다.
- 화주는 여러 대리점의 조건을 한 화면에서 비교한다.
- 최저가가 항상 최종 선택 기준은 아니다. 서비스 조건과 평판이 함께 고려된다.
- 동일 택배사라도 대리점별 단가와 서비스 조건은 다를 수 있다.
- 계약 요청 정보가 구체적일수록 대리점은 더 정확한 단가를 제안할 수 있다.
- 계약 요청이 여러 배송 품목 라인으로 구성되면 대리점 제안도 배송 품목 라인별 단가를 가질 수 있어야 한다.
- 제안의 대표 단가는 목록과 비교 화면을 위한 요약값이며, 최종 계약 품목 단가는 제안 품목 라인별 단가를 스냅샷으로 사용한다.

### 계약 요청 유형과 일정 조건

계약 요청은 단건 계약 요청과 정기 계약 요청으로 나뉠 수 있다. 화주는 계약 요청 등록 시 계약 유형을 선택한다.

- 단건 계약 요청: 특정 기간 안에 회수하고, 특정 기간 안에 배송되기를 원하는 일회성 물량이다.
- 정기 계약 요청: 반복적으로 발생하는 물량에 대해 대리점과 조건을 맞추는 계약 요청이다.

단건 계약 요청은 다음 일정 조건을 가진다.

- 회수 희망 기간: 대리점이 물품을 회수해야 하는 날짜 범위다. 예를 들어 `2026-06-22`부터 `2026-06-26`까지처럼 저장한다.
- 배송 희망 기간: 실제 배송이 완료되기를 원하는 날짜 범위다. 회수 희망 기간과 별도로 둔다.
- 회수 희망 시간: 기존 픽업 시작/종료 시간처럼 하루 안의 회수 가능 시간대를 표현한다.

정기 계약 요청은 반복 회수 조건을 가진다.

- 반복 주기: 매주, 매월 같은 반복 단위를 표현한다.
- 매주 반복: 월요일, 수요일처럼 요일을 선택한다.
- 매월 반복: 매월 10일처럼 월중 일을 선택한다.
- 반복 회수 시간: 반복 회수일마다 적용되는 시작/종료 시간이다.
- 정기 계약도 배송 희망 기간 또는 배송 소요 기준이 필요할 수 있으나, 실제 구현 정책은 반복 배송 SLA 형태로 확장할지 별도 결정이 필요하다.

일정 검증 기준은 다음을 기본으로 한다.

- 단건 계약 요청의 회수 시작일은 회수 종료일보다 늦을 수 없다.
- 단건 계약 요청의 배송 시작일은 배송 종료일보다 늦을 수 없다.
- 배송 희망 기간은 회수 희망 기간보다 앞설 수 없다. 세부 허용 기준은 운영 정책에 따라 완화할 수 있다.
- 정기 계약 요청은 반복 주기별 필수 값이 다르다. 매주 반복은 요일이 필요하고, 매월 반복은 월중 일이 필요하다.
- 반복 회수 시간의 시작 시간은 종료 시간보다 늦을 수 없다.

### 제안 단가 조율 규칙

- 제안은 최종 계약 전 단가 조율의 기준 단위다.
- 단가 조율 화면은 프론트에서 채팅방처럼 표현할 수 있지만, 백엔드의 핵심 모델은 실시간 채팅이 아니라 `proposal` 단위 협상 이벤트 원장이다.
- 협상 이벤트는 `proposalId`별 `sequence`로 정렬한다. 같은 제안 안에서는 순번이 중복되면 안 된다.
- RDB를 원천 저장소로 사용한다. Redis나 MQ는 추후 알림, 실시간 표시, 비동기 처리 보조 수단으로 검토하되 계약 단가의 원천 상태로 두지 않는다.
- 동시에 여러 사용자가 같은 제안에 단가 조율을 요청할 수 있으므로 `proposal` 행을 짧게 잠그고 `pendingNegotiationId`, `nextSequence`를 갱신한다.
- 하나의 제안에는 응답 대기 중인 가격 조율 이벤트가 최대 1개만 존재한다. 대기 중 이벤트가 있으면 새 가격 조율, 제안 수정, 최종 계약 수락을 막는다.
- 가격 조율 이벤트를 만든 당사자는 같은 이벤트를 수락하거나 거절할 수 없다. 반드시 상대방이 응답한다.
- 가격 조율 이벤트도 배송 품목 라인별 단가를 가질 수 있다. 단일 단가는 대표값이고, 실제 확정 계약 품목 단가는 품목 라인별 조율 단가를 우선한다.
- 가격 조율이 수락되면 제안의 현재 단가, 품목별 단가, 최종 합의 단가가 갱신된다. 이후 화주가 제안을 수락하면 최종 계약은 이 합의 단가를 스냅샷으로 가진다.
- 가격 조율이 거절되면 대기 상태만 해제하고 기존 제안 단가는 유지한다.

### 화주에게 주는 가치

- 여러 대리점의 단가와 조건을 한 화면에서 비교할 수 있다.
- 더 많은 선택지를 확보해 단가와 서비스 품질을 개선할 수 있다.
- 상담, 서류, 방문 중심 계약을 온라인 계약으로 단순화한다.
- 계약 진행 속도를 높이고 비교 기준을 투명하게 만든다.

### 대리점에게 주는 가치

- 신규 화주 또는 기존 화주 물량을 확보할 경로를 제공한다.
- 플랫폼에서 화주 신청 정보가 자동으로 전달되므로 영업 기회가 늘어난다.
- 대형 대리점 위주의 기존 영업 구조보다 공정하게 경쟁할 수 있다.
- 화주의 예상 물량과 배송 조건을 사전에 확인해 효율적으로 대응할 수 있다.

### 시장에 주는 가치

- 아날로그 계약 방식을 디지털 계약 방식으로 바꾼다.
- 입찰 경쟁을 통해 가격 형성을 더 투명하게 만든다.
- 데이터 기반 계약 구조를 만들어 시장 변화에 빠르게 대응할 수 있다.
- 화주에게는 선택권을, 대리점에게는 기회를 제공해 시장 효율성을 높인다.

## 한국 택배 시장 규모

기획 자료 기준 한국은 세계적으로도 손꼽히는 택배 대국이다.

- 연간 택배 물량: 37억 건 이상
- 시장 규모: 약 25조 원에서 30조 원 수준
- 이용 빈도: 국민 1인당 연간 70건 이상이라는 표현이 있으나, 실제 통계 기준은 별도 검증이 필요하다.
- 성장 요인: 온라인 쇼핑, 이커머스, 신선식품, 새벽배송, 풀필먼트 서비스
- 물량 성장률: 매년 약 5%에서 7% 증가하는 흐름으로 설명됨

시장은 크고 성장하지만, 계약 구조는 여전히 전화와 소개 중심이다. 이 구조적 공백을 디지털 플랫폼으로 연결하는 것이 택배왕의 핵심 기회다.

## 수익 모델

택배왕의 기본 수익 모델은 계약 수수료다. 화주와 대리점이 플랫폼을 통해 계약하고 실제 발송이 발생하면, 택배 1건당 일정 비율을 수수료로 받을 수 있다.

예시:

```text
월 발송 물량: 20,000건
택배비: 건당 2,100원
택배왕 수수료: 3%

2,100원 * 3% * 20,000건 = 월 1,260,000원
```

거래 규모가 크기 때문에 작은 단위의 수수료도 누적되면 큰 매출로 이어질 수 있다.

### 추가 수익 가능성

- 화주 대상 포장재, 박스, 화물 보험 광고
- 대리점 대상 상위 노출 또는 프리미엄 노출 상품
- 결제 대행 및 정산 대행 수수료
- 택배 물량 데이터 기반 부가 서비스

## 차별성

현재 화주와 대리점을 직접 연결하는 표준 플랫폼은 아직 명확하지 않다. 시장에는 택배사 구조, 본사 배정, 대리점 개별 영업, 3PL 등 다양한 방식이 존재하지만, 전국 단위 표준 계약 플랫폼은 부족하다.

택배왕의 차별성은 다음과 같다.

- 화주와 대리점을 직접 연결한다.
- 대리점이 입찰 방식으로 조건을 제안한다.
- 화주는 조건을 비교하고 직접 선택한다.
- 플랫폼은 직접 배송하지 않고 중개에 집중한다.
- 실제 배송과 기사 운영 책임은 기존 택배사와 대리점 구조에 남긴다.

이 구조는 운영 리스크를 낮추면서도 플랫폼 수수료 기반으로 확장할 수 있는 모델이다.

## 알림 도메인

택배왕의 알림은 계약, 제안, 배송기사 계약, 추후 배송 상태 변경처럼 사용자가 놓치면 안 되는 상태 변화를 알려주는 사용자 이력이다.

알림은 단순 캐시가 아니라 읽음/안읽음 상태와 재접속 후 조회가 필요하므로 Redis 단독 저장은 적합하지 않다. 기본 저장소는 MySQL을 사용하고, Redis는 추후 안 읽은 알림 수 캐시나 실시간 pub/sub 보조 용도로 검토한다.

### 보관 정책

초기 구현 기준은 다음과 같다.

- DB에는 알림을 저장한다.
- 조회 API는 기본적으로 최근 30일 알림을 대상으로 한다.
- 오래된 알림 삭제는 별도 scheduler나 운영 정책으로 다룬다.
- 읽지 않은 알림을 30일 후 바로 삭제할지는 추후 사용자 경험 기준으로 재검토한다.

일주일 보관은 사용자가 며칠 접속하지 않았을 때 제안, 계약 확정, 배송기사 계약 같은 중요한 알림을 놓칠 수 있어 너무 짧다.

### 알림 테이블 기준

알림은 상황마다 문구와 이동 경로가 다르지만 공통 저장 구조는 같으므로 하나의 `notifications` 테이블로 관리한다. 상황 구분은 enum으로 관리한다.

권장 필드:

- `id`: 알림 식별자
- `receiver_user_id`: 알림을 받을 사용자
- `sender_user_id`: 알림을 발생시킨 사용자. 시스템 알림이면 null 가능
- `type`: 알림 상황 enum
- `title`: 알림 제목
- `message`: 알림 본문
- `link_url`: 프론트에서 이동할 URL
- `reference_type`: 관련 도메인 타입
- `reference_id`: 관련 도메인 식별자
- `read_at`: 읽은 시각. null이면 읽지 않음
- `created_at`, `updated_at`: 생성/수정 시각

`title`과 `message`는 알림 생성 시점에 최종 문구로 저장한다. 문구 정책이 나중에 바뀌더라도 과거 알림은 당시 사용자에게 보여준 의미를 유지해야 하기 때문이다.

### 알림 타입 후보

계약 요청과 제안 흐름:

- `CONTRACT_REQUEST_CREATED`: 새 계약 요청 등록
- `PROPOSAL_SUBMITTED`: 대리점이 제안 제출
- `PROPOSAL_UPDATED`: 대리점이 제안 수정
- `PROPOSAL_WITHDRAWN`: 대리점이 제안 철회
- `PROPOSAL_ACCEPTED`: 화주가 제안 수락
- `PROPOSAL_REJECTED`: 제안이 선택되지 않음
- `CONTRACT_CREATED`: 최종 계약 생성
- `CONTRACT_CANCELED`: 계약 취소

대리점과 배송기사 계약 흐름:

- `DELIVER_CONTRACT_REQUESTED`: 배송기사 계약 요청
- `DELIVER_CONTRACT_ACCEPTED`: 배송기사 계약 수락
- `DELIVER_CONTRACT_REJECTED`: 배송기사 계약 거절

추후 실제 배송 상태 관리가 추가되면 다음 타입을 확장 후보로 둔다.

- `SHIPMENT_CREATED`
- `SHIPMENT_PICKUP_ASSIGNED`
- `SHIPMENT_PICKED_UP`
- `SHIPMENT_IN_TRANSIT`
- `SHIPMENT_OUT_FOR_DELIVERY`
- `SHIPMENT_DELIVERED`
- `SHIPMENT_RETURN_REQUESTED`
- `SHIPMENT_RETURNED`
- `SHIPMENT_CANCELED`

### 알림 참조 타입 후보

알림 클릭 시 어떤 도메인 화면으로 이동할지 알 수 있도록 `reference_type`과 `reference_id`를 함께 저장한다.

- `CONTRACT_REQUEST`
- `PROPOSAL`
- `CONTRACT`
- `DELIVER_CONTRACT`
- 추후 배송 실행 도메인 추가 시 `SHIPMENT`

### 우선 기록할 상황

1. 대리점이 계약 요청에 제안 제출
   - 수신자: 화주
   - 예: 새 제안이 도착했습니다.
2. 대리점이 제안 수정
   - 수신자: 화주
   - 예: 제안 조건이 수정되었습니다.
3. 대리점이 제안 철회
   - 수신자: 화주
   - 예: 제안이 철회되었습니다.
4. 화주가 제안 수락
   - 수신자: 수락된 제안을 낸 대리점
   - 예: 제안이 수락되어 계약이 생성되었습니다.
5. 화주가 다른 제안을 선택해 나머지 제안이 거절됨
   - 수신자: 선택되지 않은 제안을 낸 대리점
   - 예: 제안이 선택되지 않았습니다.
6. 최종 계약 생성
   - 수신자: 화주, 대리점
   - 예: 계약이 확정되었습니다.
7. 배송기사 계약 생성, 수락, 거절
   - 수신자: 대리점 또는 배송기사

### 구현 방향

초기 구현은 이벤트 브로커 없이 단순하게 시작한다.

```text
도메인 서비스
-> NotificationService
-> NotificationRepository
-> MySQL notifications
```

다만 계약, 제안, 배송기사 계약 서비스 안에 알림 문구 생성 로직이 흩어지지 않도록 알림 생성 책임은 `NotificationService`에 모은다.

초기 API 후보:

```text
GET /api/v1/notifications/me
GET /api/v1/notifications/me/unread-count
PUT /api/v1/notifications/{notificationId}/read
PUT /api/v1/notifications/me/read-all
```

실시간 알림, SSE/WebSocket, Redis unread count cache는 2차 확장으로 둔다.

## 도메인 전제와 확인 필요 사항

- `택배왕`은 기획 자료에서 제시된 서비스명이다.
- 시장 규모, 물량, 성장률 수치는 기획 자료 기반이며 최신 공식 통계로 검증이 필요하다.
- 실제 구현 단계에서는 화주, 대리점, 계약, 입찰, 배송 조건, 리뷰/평판, 정산을 주요 도메인으로 볼 수 있다.
- 택배사 본사와 대리점 간 계약 권한, 플랫폼 중개 수수료의 법적 성격, 전자계약 효력은 별도 검토가 필요하다.
